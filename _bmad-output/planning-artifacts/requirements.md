# Requirements Clarification

This document restates the prototype requirements with an explicit distinction between real implementation components and simulated environment components. It is a clarification document only.

## Functional Interpretation

### Requirements that must be implemented as real executable behavior

- edges must execute their own acquisition, publication, consumption, and local replicated-state logic
- MQTT communication must occur through a real broker and real publish/subscribe flow
- consensus must execute as a real validation component
- comparison must execute as a real service using consensused valid state and SCADA-side state
- valid data persistence must execute as a real local storage flow
- LSTM training and inference must execute as real downstream processing
- logs, summaries, audit packages, and alerts must be produced by the running system

### Requirements satisfied through simulation or local stand-ins

- sensor readings may be generated by a simulator rather than physical hardware
- compressor/process behavior may be generated by a simulator rather than a live industrial asset
- SCADA-side state may be provided by a fake OPC UA service rather than a production SCADA platform
- cloud storage may be represented by local object storage rather than a real cloud provider

## Non-Functional Interpretation

The prototype must remain:

- local
- simple
- modular
- demonstrable
- reproducible
- explicit about what is real execution versus simulated environment

## Boundary Rule

Simulation is allowed only for the surrounding environment. It is not allowed to replace the core research execution pillars.

Therefore:

- simulation may provide upstream sensor/process inputs
- simulation may provide SCADA-side environment inputs
- simulation may represent cloud surroundings
- simulation must not replace MQTT communication
- simulation must not replace consensus execution
- simulation must not replace comparison logic
- simulation must not replace persistence flow
- simulation must not replace LSTM execution

## Architectural Reading Rule

When reading the prototype requirements, the correct interpretation is:

- real implementation components establish the academic validity of the prototype
- simulated environment components reduce infrastructure burden without changing the architectural meaning
- no clarification in this document increases scope or adds new components
